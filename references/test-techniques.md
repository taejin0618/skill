## 테스트 설계 방법론

### 1. 명세 기반 테스트 (Specification-Based Testing)

기획서를 기준으로 동등 분할, 경계값 분석, 조건 조합, 상태 전이 테스트 기법을 적용.

**활용 시점:**
- 상세 기획서나 설계 문서가 존재할 때
- Figma/Axure 디자인 문서 기반 테스트 시

### 2. 시나리오 기반 테스트 (Scenario-Based Testing)

실제 사용자 관점의 업무 흐름과 예외 상황을 시나리오로 구성.

**시나리오 유형:**

| 유형 | 설명 | 예시 |
|------|------|------|
| 정상 (Happy Path) | 목표 성공 달성 경로 | 신규 회원 → 상품 검색 → 구매 완료 |
| 예외 | 오류/중단/재시도 | 결제 중 네트워크 끊김 → 재시도 → 완료 |
| 대체 | 다른 방법으로 목표 달성 | 카드 결제 실패 → 계좌이체 → 완료 |

### 3. 요구사항 기반 테스트 (Requirements-Based Testing)

요구사항 문서 기반 테스트 케이스 도출. Jira/Confluence 활용 시 적용.

**적용 방법:**
1. 요구사항 항목(Req ID)과 테스트 케이스 매핑
2. Acceptance Criteria → 테스트 케이스 변환
3. 요구사항 변경 시 해당 테스트 케이스 업데이트

## 세부 테스트 기법

### 1. 경계값 분석 (Boundary Value Analysis)

**적용 대상:** 숫자, 날짜, 문자열 길이, 파일 크기

**테스트 값 세트:**
```
최소값-1 (무효) | 최소값 (유효) | 최소값+1 (유효)
              중간값 (유효)
최대값-1 (유효) | 최대값 (유효) | 최대값+1 (무효)
```

**예시: 비밀번호 8~20자 제한**

| 입력 | 길이 | 예상 결과 |
|------|------|----------|
| Pass123 | 7자 | ❌ 에러 |
| Pass1234 | 8자 | ✅ 성공 |
| Pass12345 | 9자 | ✅ 성공 |
| Pass1234567890123456789 | 21자 | ❌ 에러 |

### 2. 동등 분할 (Equivalence Partitioning)

**적용 대상:** 입력 값의 유형 분류

**예시: 이메일 형식 검증**

| 그룹 | 예시 | 결과 |
|------|------|------|
| 유효 - 표준 | user@example.com | ✅ |
| 유효 - 서브도메인 | user@mail.example.com | ✅ |
| 무효 - @ 없음 | userexample.com | ❌ |
| 무효 - . 없음 | user@examplecom | ❌ |
| 무효 - 공백 포함 | user @example.com | ❌ |

### 3. 상태 전환 테스팅 (State Transition Testing)

**적용 대상:** 상태가 변하는 기능 (주문, 게시글, 사용자 상태)

**예시: 주문 상태**
```
[주문생성] → 결제완료 → [결제완료] → 배송시작 → [배송중] → 배송완료 → [배송완료] → 구매확정 → [구매확정]
```

**테스트 케이스:**
- ✅ 정상 전환: 결제완료 → 배송중
- ❌ 비정상 전환: 배송중 → 주문생성 (불가해야 함)

### 4. 조건 조합 테스트 (Combination Testing)

**적용 대상:** 여러 조건이 동시에 작용하는 기능

**기법:**
- **Pairwise**: 모든 조건 쌍을 최소 1회 테스트 (효율적)
- **All Combinations**: 모든 조합 테스트 (조건 적을 때만)

**예시: 회원 등급별 할인**

| 조건 | 값 |
|------|------|
| 회원 등급 | 일반, VIP, VVIP |
| 상품 카테고리 | 전자, 의류, 식품 |
| 구매 금액 | 10만원 미만, 10~50만원, 50만원 이상 |

전체 조합: 3×3×3 = 27개 → Pairwise로 9~12개로 축소 가능
