## 기본 원칙

### ✅ Happy Path에 포함할 것

- 유효한 입력값만 사용
- 정상적인 사용자 행동
- 성공 시 UI 피드백 (성공 메시지, 로딩 인디케이터)
- 기대하는 정상 결과
- 데이터 정상 표시 및 이동

### ❌ Happy Path에 포함하지 않을 것

#### Error Case 유형 (제외)

| 유형 | 예시 |
|------|------|
| 잘못된 입력값 | 빈 값, 형식 오류, 범위 초과 |
| 권한/접근 오류 | 권한 없는 사용자, 접근 거부 |
| 중복/충돌 | 중복 데이터, 이미 존재하는 값 |
| 오류 메시지 검증 | 에러 메시지, 오류 아이콘 확인 |
| 실패 시나리오 | 저장 실패, 조회 실패, 전송 실패 |

#### Edge Case 유형 (제외)

| 유형 | 예시 |
|------|------|
| 경계값 테스트 | 최대/최소 글자수, 최댓값/최솟값 |
| 특수 입력값 | 특수문자, 이모지, 공백만, 줄바꿈만 |
| 동시성/중복 | 중복 요청, 연속 클릭, 이중 제출 |
| 브라우저 동작 | 뒤로가기, 새로고침, 탭 전환 |
| 세션/네트워크 | 세션 만료, 타임아웃, 연결 끊김, 오프라인 |
| 보안 취약점 | XSS, SQL 인젝션, 스크립트 삽입 |

## UI 요소별 정상 동작 예시

### 1. 입력란 (Input)

#### ✅ Happy Path 예시

**테스트케이스**: 유효한 이메일 입력
```
Steps:
1. 이메일 입력란에 'test@example.com' 입력
2. [저장] 버튼 클릭

Expected Result:
1. '저장되었습니다' 메시지 표시
2. 입력한 이메일이 화면에 표시
```

**테스트케이스**: 유효한 이름 입력
```
Steps:
1. 이름 입력란에 '홍길동' 입력
2. [저장] 버튼 클릭

Expected Result:
1. 저장 성공 메시지 표시
2. 이름이 목록에 추가됨
```

#### ❌ Happy Path에 포함하지 않는 예시

- 빈 값 입력 시 에러 검증
- 잘못된 이메일 형식 (예: `test@`)
- 최대 길이 초과
- 특수문자/이모지 입력

---

### 2. 버튼 (Button)

#### ✅ Happy Path 예시

**테스트케이스**: 검색 버튼 클릭 정상 동작
```
Steps:
1. 검색어 입력란에 '홍길동' 입력
2. [검색] 버튼 클릭

Expected Result:
1. 로딩 인디케이터 표시
2. 검색 결과 목록 표시
3. 검색어에 맞는 데이터만 표시
```

**테스트케이스**: 저장 버튼 클릭 정상 동작
```
Steps:
1. 모든 필수 입력란에 유효한 값 입력
2. [저장] 버튼 클릭

Expected Result:
1. '저장되었습니다' 성공 메시지 표시
2. 목록 페이지로 이동
3. 저장한 데이터가 목록에 표시
```

#### ❌ Happy Path에 포함하지 않는 예시

- 필수 입력 누락 시 버튼 동작
- 더블클릭 중복 요청 방지
- 권한 없는 사용자의 버튼 클릭

### 3. 드롭다운/셀렉트박스 (Select)

#### ✅ Happy Path 예시

**테스트케이스**: 상태 선택 정상 동작
```
Steps:
1. 상태 드롭다운 클릭
2. '승인' 선택
3. [저장] 버튼 클릭

Expected Result:
1. 선택한 '승인' 값이 드롭다운에 표시
2. 저장 후 상태가 '승인'으로 저장됨
```

**테스트케이스**: 부서 필터 선택
```
Steps:
1. 부서 필터 드롭다운 클릭
2. '개발팀' 선택

Expected Result:
1. 목록이 '개발팀'으로 필터링됨
2. 다른 부서 데이터는 표시되지 않음
```

### 4. 체크박스 (Checkbox)

#### ✅ Happy Path 예시

**테스트케이스**: 전체 선택 체크박스
```
Steps:
1. 목록에 데이터 10건 표시
2. 전체 선택 체크박스 클릭

Expected Result:
1. 모든 항목의 체크박스가 선택됨
2. 선택된 항목 개수 '10개 선택' 표시
```

**테스트케이스**: 개별 선택 체크박스
```
Steps:
1. 첫 번째 항목 체크박스 선택
2. 세 번째 항목 체크박스 선택

Expected Result:
1. 선택된 2개 항목만 체크 표시
2. 선택된 항목 개수 '2개 선택' 표시
```

#### ✅ Happy Path 예시

**테스트케이스**: 페이지 이동 정상 동작
```
Preconditions:
- 데이터 50건 이상 존재 (페이지당 10건 표시)

Steps:
1. 회원 목록 화면 접속 (1페이지 표시)

3. '다음' 버튼 클릭
4. '이전' 버튼 클릭

Expected Result:
1. 각 페이지 클릭 시 해당 페이지 데이터 표시
2. 페이지 번호 하이라이트 정상 변경
3. 전체 건수 정보 정확히 표시 (예: '총 50건')
```

- 파일 크기 초과

## 성공 시나리오 예시

### 로그인 플로우

**테스트케이스**: 유효한 계정으로 로그인
```
Preconditions:
1. 로그인 권한: 일반 사용자
2. 접근 경로: 메인 > 로그인
3. 테스트 데이터: test@example.com / Pass1234!
4. 사전 상태: 회원가입 완료된 계정 존재

Steps:
1. 로그인 페이지 접속 (URL: /login)
2. 이메일 입력란에 'test@example.com' 입력
3. 비밀번호 입력란에 'Pass1234!' 입력
4. [로그인] 버튼 클릭

Expected Result:
1. 대시보드 페이지로 이동 (URL: /dashboard)
2. 우측 상단에 'test@example.com' 표시
3. '로그인 성공' 토스트 메시지 표시
4. 로딩 인디케이터 정상 표시 후 사라짐

Priority: Highest
```

---

### 회원 등록 플로우

**테스트케이스**: 신규 회원 등록 성공
```
Preconditions:
1. 로그인 권한: 관리자
2. 접근 경로: 메인 > 회원관리 > 회원 등록
3. 테스트 데이터:
   - 이름: 홍길동
   - 이메일: hong@example.com
   - 부서: 개발팀
4. 사전 상태: 없음

Steps:
1. 회원 등록 화면 접속
2. 이름 입력란에 '홍길동' 입력
3. 이메일 입력란에 'hong@example.com' 입력
4. 부서 드롭다운에서 '개발팀' 선택
5. [저장] 버튼 클릭

Expected Result:
1. 로딩 인디케이터 표시
2. '회원이 등록되었습니다' 성공 메시지 표시
3. 회원 목록 페이지로 이동
4. 목록에 '홍길동' 회원 표시

Priority: High
```

---

### 상품 주문 플로우

**테스트케이스**: 상품 선택 후 주문 완료
```
Preconditions:
1. 로그인 권한: 일반 사용자
2. 접근 경로: 메인 > 상품 목록
3. 테스트 데이터: 상품명 '노트북', 수량 2개
4. 사전 상태: 상품 재고 10개 이상

Steps:
1. 상품 목록에서 '노트북' 선택
2. 수량 '2' 입력
3. [장바구니 담기] 버튼 클릭
4. [주문하기] 버튼 클릭
5. 주문 확인 화면에서 [결제] 버튼 클릭

Expected Result:
1. 각 단계마다 성공 메시지 표시
2. 장바구니에 상품 추가됨
3. 주문 완료 페이지로 이동
4. '주문이 완료되었습니다' 메시지 표시
5. 주문 번호 표시

Priority: Highest
```

## 요약

### Happy Path 테스트의 핵심

1. **유효한 입력값만 사용**: 정상 범위 내 데이터
2. **성공 시나리오 집중**: 기대하는 정상 플로우만 검증
3. **UX 피드백 포함**: 성공 메시지, 로딩 표시 확인
4. **측정 가능한 결과**: 구체적인 메시지 텍스트, UI 상태 명시

### 제외할 항목

- 입력 검증 실패 케이스
- 에러 메시지 검증
- 경계값 초과/미달
- 엣지 케이스 (특수문자, 이모지, 중복 요청)
- 브라우저 동작 테스트 (뒤로가기, 새로고침)
