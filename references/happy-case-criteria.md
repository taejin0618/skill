# Happy Case 분류 기준 상세 가이드 (강화 버전)

AI가 테스트케이스를 분류할 때 참조하는 상세 기준이다.
판단 순서: Expected Result → Title+Section+Hierarchy → Steps → Edge 키워드(전체 필드) → Edge 정규식(전체 필드)

---

## 1. Happy Case 포함 기준

아래 3가지 필드를 종합하여 판단한다. 하나의 필드만으로 판단하지 않는다.

### 1-1. Title 기준 (포함 키워드)

| 카테고리 | 키워드 | 예시 Title |
|----------|--------|-----------|
| 긍정 수식어 | 유효한, 정상적인, 올바른 | "유효한 이메일로 가입" |
| 성공 상태 | 성공, 완료, 정상, 확인 | "로그인 성공 후 대시보드 이동" |
| 일반 동작 | 조회, 검색, 등록, 저장, 수정, 삭제 | "회원 정보 정상 저장" |
| UI 동작 | 클릭, 선택, 입력, 이동, 표시, 열기, 닫기 | "검색 버튼 클릭 정상 동작" |
| 파일 동작 | 다운로드, 업로드, 내보내기, 가져오기 | "엑셀 다운로드 정상 동작" |

### 1-2. Steps 기준 (Happy Case 특성)

- 구체적인 유효 입력값 사용 (예: `'test@example.com'`, `'홍길동'`, `'2025-01-15'`)
- 정상적인 순서로 진행 (접속 → 입력 → 버튼 클릭 → 결과 확인)
- 표준 사용자 행동 패턴 준수
- 1건 이상의 유효 데이터 사용

### 1-3. Expected Result 기준 (반드시 성공 결과)

| 성공 패턴 | 예시 |
|-----------|------|
| 성공/완료 메시지 | `'저장되었습니다'`, `'등록 완료'`, `'처리되었습니다'`, `'삭제되었습니다'` |
| 페이지 이동 | `대시보드로 이동`, `목록 페이지로 이동`, `URL 변경됨` |
| 데이터 표시 | `목록에 추가됨`, `화면에 표시됨`, `검색 결과 N건 표시` |
| 로딩 완료 | `로딩 인디케이터 표시 후 사라짐` |
| 정상 상태 | `버튼 활성화됨`, `체크박스 선택됨`, `탭 전환됨` |

## 2. Error Case 제외 기준

다음 중 하나라도 해당되면 EXCLUDE 처리한다.

### 2-1. Title 제외 키워드

| 키워드 유형 | 키워드 목록 |
|-------------|------------|
| 오류/실패 명시 | 오류, 에러, 실패, error, fail |
| 부정 입력 수식어 | 잘못된, 유효하지 않은, 올바르지 않은, 틀린 |
| 빈 값 | 빈 값, 빈칸, 미입력, 입력하지 않고, 입력 없이, 미선택 |
| 거부/차단 | 접근 거부, 권한 없음, 권한없음, 차단, 금지 |
| 중복 | 중복, 이미 사용 중, 이미 존재, 재시도 |
| 경계값 | 최댓값, 최솟값, 경계값, 경계 |
| 비정상 상태 | 불일치, 초과, 미만, 부족, 만료, 타임아웃, 비정상, 예외, 누락, 위반 |

### 2-2. Expected Result 제외 패턴

| 패턴 카테고리 | 예시 |
|------|------|
| 오류 메시지 | `'올바르지 않습니다'`, `'오류가 발생했습니다'`, `'다시 시도해주세요'` |
| 필수 입력 오류 | `'필수 입력'`, `'입력해주세요'`, `'필수 항목'` |
| 시각적 오류 표시 | `빨간색 테두리`, `붉은 테두리`, `경고 아이콘`, `오류 아이콘` |
| 접근 차단 | `403`, `404`, `500`, `접근 불가`, `페이지를 찾을 수 없습니다` |
| 재입력 요구 | `재입력`, `다시 입력`, `수정 후 재시도` |
| 거부/실패 | `실패`, `거부`, `유효하지 않`, `경고 메시지` |

### 2-3. Steps 제외 패턴

| 패턴 | 예시 Steps |
|------|-----------|
| 빈 값 입력 | "이메일 입력란을 **비워두고**", "**아무것도 입력하지 않고**" |
| 잘못된 형식 | "**잘못된 이메일 형식**으로 입력", "숫자 아닌 값으로" |
| 범위 초과 | "**최대 글자수 초과** 입력", "**100자 이상** 입력" |
| 비정상 접근 | "**권한 없는 상태**로 메뉴 접근", "**로그아웃 상태**에서 URL 직접 접속" |
| 특수 입력 | "`<script>` 입력", "`' OR '1'='1` 입력", "이모지 입력" |
| 존재하지 않는 대상 | "**존재하지 않는 계정**으로 로그인" |
| 만료된 토큰/세션 | "**만료된 토큰**으로 API 호출", "**만료된 세션**에서 작업" |
| 삭제된 데이터 | "**삭제된 계정**으로 접근", "**삭제된 파일** 다운로드 시도" |

## 3. Edge Case 제외 기준

### 3-1. Edge Case 정의

경계값, 예외 상황, 비정상 상태에서의 동작 테스트 케이스

### 3-2. Edge Case 제외 키워드/패턴

| 카테고리 | 키워드/패턴 | 적용 필드 |
|----------|------------|---------|
| 경계값 (키워드) | 최댓값, 최솟값, 경계값, 경계, 최대 글자수, 자 초과, 자 미만 | 전체 필드 (Section, Hierarchy 포함) |
| 경계값 (정규식) | N자 초과/미만/이상/이하, 딱 N자, 최대 N자, 최소 N자, N건/개/회 초과/미만 | 전체 필드 (Section, Hierarchy 포함) |
| 특수 데이터 | 특수문자, 이모지, 한자, 아랍어, 공백만, 줄바꿈만 | 전체 필드 (Section, Hierarchy 포함) |
| 비정상 상태 | 데이터 없음, 0건, 빈 목록, 연결 끊김 | 전체 필드 (Section, Hierarchy 포함) |
| 동시성 | 동시에, 중복 요청, 연속 클릭, 이중 제출 | 전체 필드 (Section, Hierarchy 포함) |
| 브라우저 동작 | 뒤로가기, 새로고침 후, 브라우저 닫기, 탭 전환 | 전체 필드 (Section, Hierarchy 포함) |
| 세션/타임아웃 | 세션 만료, 타임아웃, 토큰 만료 | 전체 필드 (Section, Hierarchy 포함) |
| 네트워크 | 네트워크 끊김, 오프라인 | 전체 필드 (Section, Hierarchy 포함) |
| 보안 | XSS, SQL, 인젝션, 스크립트 | 전체 필드 (Section, Hierarchy 포함) |

> ⚠️ Edge Case 키워드는 **Section, Section Hierarchy, Title, Steps, Expected Result 5개 필드 전체**에서 감지한다.
> Expected Result에 경계값 조건이나 엣지 상황 묘사가 있어도 즉시 EXCLUDE 처리한다.

## 4. 판단 우선순위 (체크 순서)

```
Step 1: Expected Result 확인 (오류/에러/실패 패턴)
  └─ '오류', '에러', '실패', '올바르지 않습니다', 빨간 테두리, '거부', '경고 메시지' 등 → EXCLUDE

Step 2: Title + Section + Section Hierarchy 확인 (오류/비정상 키워드)
  └─ '잘못된', '실패', '오류', '빈 값', '중복', '경계', '초과', '만료', '누락' 등 → EXCLUDE
  └─ Section명이 "오류 처리", "에러 케이스" 등인 경우도 EXCLUDE

Step 3: Steps 확인 (비정상 입력 패턴)
  └─ '비워두고', '잘못된 값', '특수문자', '동시에', '존재하지 않는', '만료된', '삭제된' 등 → EXCLUDE

Step 4: Edge Case 키워드 확인 (전체 필드: Title + Steps + Expected + Section + Section Hierarchy)
  └─ '한자', '아랍어', '이모지', '동시에', '세션 만료', '뒤로가기', XSS, SQL 등 → EXCLUDE
  └─ Section Hierarchy 경로에 엣지케이스 관련 단어가 있어도 EXCLUDE

Step 5: Edge Case 경계값 정규식 확인 (전체 필드)
  └─ "N자 초과/미만", "딱 N자", "최대 N자", "N건 초과" 등 숫자+단위 패턴 → EXCLUDE

Step 6: 위 5단계 해당 없음 → HAPPY
```

---

## 5. Section 행 처리 규칙

```
규칙 1: Title이 비어있는 행 = Section 헤더 행
규칙 2: Section 행 자체는 HAPPY/EXCLUDE 분류 대상 아님
규칙 3: 해당 Section 내에 Happy Case가 1개 이상 있으면 Section 행 포함
규칙 4: 해당 Section 내 모든 케이스가 EXCLUDE이면 Section 행도 제외
규칙 5: Section Hierarchy 경로상의 모든 상위 Section 행도 포함
```

## 6. 애매한 케이스 처리 기준

| 유형 | 판단 | 근거 |
|------|------|------|
| 삭제 확인 다이얼로그 후 삭제 성공 | **HAPPY** | 정상 UX 플로우, 성공 메시지로 종료 |
| 취소 버튼 클릭 → 이전 화면 이동 | **HAPPY** | 정상 사용자 행동 |
| 관리자 권한으로 설정 메뉴 접근 | **HAPPY** | 유효 권한 사용, 성공 접근 |
| 전체 선택 체크박스로 일괄 선택 | **HAPPY** | 정상 기능 동작 |
| 비밀번호 변경 성공 | **HAPPY** | 유효 값 사용, 성공 메시지 |
| 0건 검색 결과 표시 | **EXCLUDE** | 경계 상황 (빈 상태) |
| 빈 목록 "데이터 없음" 메시지 | **EXCLUDE** | 예외 상태 |
| 경계값(최대 N자) 정상 저장 | **EXCLUDE** | Title/Steps에 '최대', '경계' 키워드 |
| 세션 만료 후 재로그인 | **EXCLUDE** | 비정상 상태 진입 |
| 만료된 토큰으로 API 호출 | **EXCLUDE** | 비정상 입력값 사용 |
| 존재하지 않는 계정 접근 | **EXCLUDE** | 비정상 데이터 사용 |
| 오프라인 상태에서 동작 | **EXCLUDE** | 비정상 네트워크 환경 |

## 7. 행별 분류 체크리스트 (빠른 판단)

```
□ Expected Result:
  - '오류/에러/실패/거부/경고 메시지' 패턴? → EXCLUDE
  - '성공/완료/이동/표시' 패턴? → HAPPY 방향

□ Title + Section + Section Hierarchy:
  - '잘못된/오류/실패/빈 값/중복/경계/초과/만료/누락' 키워드? → EXCLUDE
  - Section명이 "오류 처리", "에러 케이스" 등? → EXCLUDE
  - 위 키워드 없음? → 다음 단계

□ Steps:
  - '비워두고/잘못된 값/특수문자/동시에/존재하지 않는/만료된/삭제된' 패턴? → EXCLUDE
  - 유효한 구체값 사용? → HAPPY 방향

□ Edge Case 키워드 (Section, Hierarchy, Title, Steps, Expected 전체 필드):
  - '한자/아랍어/이모지/동시에/세션 만료/뒤로가기/XSS/SQL' 등? → EXCLUDE

□ Edge Case 경계값 정규식 (전체 필드):
  - 'N자 초과/미만/이상/이하', '딱 N자', '최대 N자', 'N건 초과' 등? → EXCLUDE

□ 종합:
  - 5가지 모두 이상 없음 → HAPPY
  - 하나라도 Exclude 시그널 → EXCLUDE
```
