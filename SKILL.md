# 기획서 분석 파이프라인 스킬 v3

기획서(Confluence/PPTX/PDF)를 체계적으로 분석하고, Figma 디자인과 교차 검증하여
빠진 내용, 서로 맞지 않는 부분, 불명확한 정의를 찾아낸다.

## 핵심 원칙

1. **Confluence가 기준 문서(PRD)**: Confluence 기획서를 정본(Source of Truth)으로 취급. PPTX/PDF는 보조 자료
2. **근거 기반 지적**: 모든 이슈에 "어디서 발견했는지"와 "왜 문제인지"를 명시
3. **보고용 출력**: 이해관계자(PO, 개발자, 디자이너)가 바로 읽고 판단할 수 있는 쉬운 표현

## 작성 톤 & 용어 가이드

출력물은 이해관계자 보고용이므로 다음 규칙을 따른다:

| 피해야 할 표현 | 대신 사용할 표현 |
|---------------|----------------|
| 상태충돌 | 앞뒤가 맞지 않는 흐름 |
| 정의누락 | 설명이 빠진 항목 |
| 명칭불일치 | 이름이 다르게 쓰인 항목 |
| 엔티티 | 주요 대상 (사용자, 주문 등) |
| 상태 전이 | 단계별 흐름 변화 |
| 트리거 | 실행 조건 / 발생 시점 |
| 모호성 점수 | 명확도 점수 |
| globalModel | 서비스 구조 요약 |
| Cross-Chunk | 섹션 간 비교 |
| Edge Case | 예외 상황 / 놓치기 쉬운 시나리오 |

**문장 스타일:**
- "~가 정의되지 않았습니다" → "~에 대한 설명이 빠져 있습니다"
- "상태 전이 모순 발견" → "A 단계에서 B 단계로 넘어가는 조건이 앞뒤가 맞지 않습니다"
- 전문 용어를 써야 할 때는 괄호로 쉬운 설명 병기: "Variant(같은 화면의 다른 상태)"

## 심각도 기준

모든 이슈에 아래 기준으로 심각도를 부여한다.

| 심각도 | 아이콘 | 기준 | 예시 | 조치 시한 |
|--------|--------|------|------|-----------|
| **Critical** | 🔴 | 이대로 개발하면 서비스 장애나 데이터 손실이 발생할 수 있음 | 결제 금액 계산 규칙이 서로 모순, 개인정보 처리 기준 누락 | 개발 착수 전 반드시 해결 |
| **Major** | 🟠 | 핵심 기능이 의도와 다르게 동작할 수 있음 | Confluence와 Figma에서 버튼 동작이 다르게 정의됨, 핵심 화면 누락 | 개발 착수 전 해결 권장 |
| **Minor** | 🟡 | 사용자 경험에 혼란을 줄 수 있으나 기능 동작에는 영향 없음 | 버튼 이름이 문서마다 다름, 안내 문구 불일치 | 개발 중 수정 가능 |
| **Info** | 🟢 | 개선하면 좋지만 당장 문제는 없음 | 용어 표기 통일 권장, 부가 설명 추가 권장 | 여유 있을 때 반영 |

**심각도 판단 보조 질문:**
- "이 문제를 모르고 개발하면 어떤 일이 벌어지는가?" → 서비스 장애 = Critical, 기능 오류 = Major
- "사용자가 직접 겪는 문제인가?" → 예 = Major 이상, 아니오 = Minor 이하
- "수정하지 않아도 서비스가 돌아가는가?" → 예 = Minor/Info, 아니오 = Critical/Major

## PHASE 0: 입력 수집 — 다중 소스 통합

### 입력 유형 판별 및 수집 방법

| 입력 유형 | 역할 | 판별 기준 | 수집 도구 |
|-----------|------|-----------|-----------|
| **Confluence** | 📌 **기준 문서 (PRD)** | `atlassian.net` URL | Atlassian MCP |
| **PPTX** | 보조 문서 | `.pptx` 업로드 | `pptx` 스킬 |
| **PDF** | 보조 문서 | `.pdf` 업로드 | `pdf-reading` 스킬 |
| **Figma** | 디자인 검증용 | `figma.com` URL | Figma MCP |
| **스크린샷** | 참고 자료 | 이미지 업로드 | 이미지 분석 |

> **중요**: Confluence 문서가 기준(정본)이다. PPTX/PDF와 Confluence 내용이 다르면 Confluence를 기준으로 판단하고, PPTX/PDF 쪽에 "Confluence 기준과 다름"으로 표기한다.

### Confluence 연동 상세

**1단계: Atlassian MCP로 페이지 fetch**
- Confluence 페이지 URL로 콘텐츠 조회
- 본문(body), 제목, 라벨, 첨부파일 목록 추출

**2단계: 하위 페이지 탐색 (선택)**
- 사용자에게 "하위 페이지도 분석할까요?" 확인
- 포함 시 하위 페이지도 순차 fetch → 하나의 분석 단위로 병합
- 각 페이지에 번호 부여: `[CFP-1] 메인`, `[CFP-2] 하위1`

**3단계: Notion 보조 검색**
- Notion MCP의 `notion-search` 도구로 Confluence 연동 소스 추가 검색 가능

### Figma 연동 상세

**지원 URL:**
```
https://www.figma.com/design/{fileKey}/{fileName}
https://www.figma.com/file/{fileKey}/{fileName}
https://www.figma.com/design/{fileKey}/{fileName}?node-id={nodeId}
```

**Figma MCP 활용 절차:**

| 단계 | Figma MCP 도구 | 가져오는 내용 |
|------|----------------|-------------|
| 1. 파일 구조 조회 | `get_file` | 파일명, 페이지 목록, 수정일 |
| 2. 프레임 탐색 | `get_file` (노드 지정) | 프레임(화면) 구조, 컴포넌트 트리 |
| 3. 상세 조회 | `get_file_node` (node-id) | 텍스트, 버튼 라벨, 입력 필드 |
| 4. 디자인 속성 | 노드 분석 | 색상, 글꼴, 간격 |

**node-id가 URL에 있으면** 해당 프레임만 집중 분석, 없으면 전체 페이지 순회.

**Figma에서 추출하는 항목:**

| 추출 항목 | Confluence와 비교할 대상 |
|-----------|------------------------|
| 버튼·라벨 텍스트 | 기획서에 정의된 명칭 |
| 화면(프레임) 목록 | 기획서에 기술된 화면 목록 |
| 입력 필드 (텍스트박스, 드롭다운 등) | 기획서의 필드 정의 |
| 상태 변형 (Variant) | 기획서의 상태별 화면 정의 |
| 화면 연결 (프로토타입) | 기획서의 화면 전환 흐름 |

## PHASE 1: 전처리

### 1-1. extractText() — 텍스트 추출

| 입력 유형 | 추출 방법 | 소스 태그 |
|-----------|-----------|-----------|
| Confluence | Atlassian MCP → 본문 텍스트 | `[CFP-{N}]` |
| PPTX | 슬라이드별 (제목, 본문, 노트) | `[PPT-{N}]` |
| PDF | 페이지별 (이미지면 OCR) | `[PDF-{N}]` |

- Confluence 매크로(expand, panel, status, 표, 코드블록)도 콘텐츠로 변환
- 각 소스에 **태그** 부여하여 이슈 위치 추적

### 1-2. preprocessExtractedText() — 분석 대상 선별

**제외 대상:** 표지, 목차, 참고문헌, 부록, 빈 페이지, 변경 이력, 댓글

결과: **functionalSources** (실제 분석할 소스 태그 리스트)

### 1-3. chunkAnalysisText() — 청크 분할

- **18,000자** 단위로 나눔
- 한 소스(페이지/섹션)가 두 청크에 걸치지 않도록 분할
- 청크별로 포함된 소스 태그 기록

## PHASE 2: 청크 반복 분석 (×N)

### PASS 1 — extractChunkModel() : 서비스 구조 추출

기획서에서 **서비스가 어떻게 구성되어 있는지** 핵심 요소를 뽑아낸다.

| 추출 항목 | 쉬운 설명 | 예시 |
|-----------|-----------|------|
| **주요 대상** | 서비스에서 다루는 핵심 요소 | 사용자, 주문, 상품, 쿠폰 |
| **단계/상태** | 각 대상이 거치는 단계 | 주문: 대기 → 결제완료 → 배송중 → 완료 |
| **기능/행동** | 단계를 바꾸는 사용자 행동 | 결제하기, 취소하기, 환불요청 |
| **규칙** | 서비스 운영 규칙·제약 | "쿠폰은 1회만 사용 가능" |
| **사용자 유형** | 서비스를 쓰는 사람의 종류 | 일반회원, 관리자, 판매자 |
| **화면** | 사용자가 보는 화면 목록 | 로그인, 주문 상세, 결제 완료 |

결과를 **서비스 구조 요약**에 누적. 청크 2부터는 이전 결과를 참조하면서 분석.

### PASS 2a — findLocalChunkIssues() : 기획서 내부 문제 찾기

| 이슈 유형 | 쉬운 설명 | 심각도 |
|-----------|-----------|--------|
| 앞뒤가 맞지 않는 흐름 | 같은 기능인데 A에서는 이렇게, B에서는 저렇게 동작한다고 써있음 | 🔴 Critical |
| 설명이 빠진 항목 | 기획서에 이름만 나오고 구체적인 설명이 없음 | 🟡 Minor |
| 이름이 다르게 쓰인 항목 | 같은 걸 "주문취소"로도, "주문 취소하기"로도 부름 | 🟡 Minor |
| 조건이 불분명한 분기 | "상황에 따라 처리"처럼 구체적 기준이 없음 | 🟠 Major |
| 누가 하는 건지 불명확 | 기능은 있는데 관리자가 하는 건지 사용자가 하는 건지 안 나옴 | 🟡 Minor |

### PASS 2b — findChunkEdges() : 놓치기 쉬운 시나리오 찾기

`references/edge-analysis-guide.md` 참조하여 분석.

**분석 관점:** 경계값, 동시 접근, 오류 상황, 권한 겹침, 데이터 없는 경우, 외부 연동

**명확도 점수:** 소스별 0~10점 (10 = 매우 불명확)

| 점수 | 판정 | 의미 |
|------|------|------|
| 0~3 | ✅ 명확 | 주요 동작과 예외 상황이 잘 정리되어 있음 |
| 4~6 | ⚠️ 보완 권장 | 일부 상황에 대한 설명이 부족함 |
| 7~10 | 🚨 확인 필요 | 읽는 사람에 따라 다르게 해석할 수 있음 |

## PHASE 3: 조건부 실행

### PASS 2c — findCrossSourceIssues() `[조건: 소스 2개 이상]`

**목적**: Confluence(기준 문서)와 다른 소스(PPTX/PDF) 사이에 내용이 다른 부분 찾기

| 검증 항목 | 설명 |
|-----------|------|
| 같은 용어, 다른 뜻 | Confluence에서는 A라고 했는데 PPTX에서는 B라고 정의 |
| 규칙 충돌 | Confluence 규칙과 PPTX 규칙이 서로 모순 |
| 흐름 단절 | 한 문서의 결과가 다른 문서의 시작 조건과 안 맞음 |
| 내용 중복 | 같은 기능이 여러 문서에서 미묘하게 다르게 기술됨 |

> **판정 기준**: Confluence가 정본. 다른 소스가 다르면 "PPTX/PDF 쪽이 Confluence 기준과 다름"으로 표기.

### PASS 2d — runDesignCrossCheck() `[조건: Figma URL 제공 시]`

**목적**: Confluence 기획서와 Figma 디자인이 일치하는지 검증

**Figma MCP 호출 절차:**
```
1. Figma URL에서 fileKey 추출
2. Figma MCP get_file로 파일 전체 구조 조회
3. 화면별로 get_file_node 호출하여 텍스트·컴포넌트·연결 정보 수집
4. 서비스 구조 요약의 화면/기능/상태와 하나씩 대조
```

**Confluence ↔ Figma 대조 항목:**

| # | 검증 항목 | Confluence (기준) | Figma (비교) | 심각도 |
|---|-----------|-------------------|-------------|--------|
| 1 | 화면 목록 | 기획서에 정의된 화면 | Figma에 있는 프레임 | 🔴 누락 시 Critical |
| 2 | 화면 이동 흐름 | 기획서의 "A화면 → B화면" 정의 | Figma 프로토타입 연결 | 🟠 불일치 시 Major |
| 3 | 버튼/라벨 문구 | 기획서에 쓴 명칭 | Figma 텍스트 노드 | 🟡 불일치 시 Minor |
| 4 | 입력 필드 | 기획서에 정의한 입력 항목 | Figma 폼 컴포넌트 | 🟠 누락 시 Major |
| 5 | 상태별 화면 | 기획서의 상태 정의 (로딩, 성공, 실패 등) | Figma Variant(같은 화면의 다른 상태) | 🟡 누락 시 Minor |
| 6 | 오류 안내 화면 | 기획서의 오류 시나리오 | Figma 오류 화면 프레임 | 🟠 누락 시 Major |
| 7 | 빈 화면 (데이터 없을 때) | 기획서의 빈 상태 정의 | Figma 빈 상태 프레임 | 🟡 누락 시 Minor |
| 8 | 사용자 유형별 화면 차이 | 기획서의 역할별 기능 차이 | Figma 역할별 화면 분기 | 🟢 누락 시 Info |

**디자인 대조 결과 출력 형식:**

```markdown
## Confluence ↔ Figma 대조 결과

### 불일치 항목 요약
| # | 항목 | Confluence 기획서 | Figma 디자인 | 심각도 | 필요 조치 |
|---|------|-------------------|-------------|--------|-----------|
| 1 | 결제 버튼 문구 | "결제하기" (CFP-5) | "구매하기" | 🟡 Minor | 명칭 통일 필요 |
| 2 | 비밀번호 찾기 화면 | 정의됨 (CFP-12) | 프레임 없음 | 🔴 Critical | 디자인 추가 필요 |

### Confluence에만 있는 화면 (= Figma에 빠진 화면)
| 화면명 | Confluence 위치 | 심각도 |
|--------|----------------|--------|
| 비밀번호 찾기 | CFP-12 | 🔴 디자인 누락 |

### Figma에만 있는 화면 (= 기획서에 안 나온 화면)
| 프레임명 | Figma 위치 | 심각도 |
|----------|-----------|--------|
| Onboarding-v2 | Page: Main > Onboarding | ⚠️ 기획서 추가 필요 |
```
